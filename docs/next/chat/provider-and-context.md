# Provider adapters and future context

## 1. Provider boundary

Acorn must own a narrow model-provider contract. It should be large enough to normalize generation,
attachments, models, usage, cancellation, and errors, but not become a lowest-common-denominator copy
of every provider API.

```ts
type ChatProviderId = string

type ChatModelCapability = {
  textInput: true
  textOutput: true
  imageInput: boolean
  pdfInput: boolean
  textFileInput: boolean
  streaming: boolean
}

type ChatModelDescriptor = {
  providerId: ChatProviderId
  id: string
  label: string
  description?: string
  capabilities: ChatModelCapability
  contextWindow?: number
  maxOutputTokens?: number
  recommended?: boolean
  deprecated?: boolean
}

type CanonicalChatContent =
  | { kind: 'text'; text: string }
  | { kind: 'image'; mediaType: 'image/jpeg' | 'image/png' | 'image/gif' | 'image/webp'; bytes: Uint8Array }
  | { kind: 'pdf'; mediaType: 'application/pdf'; filename: string; bytes: Uint8Array }

type CanonicalChatMessage = {
  id: string
  role: 'user' | 'assistant'
  content: CanonicalChatContent[]
}

type CanonicalChatRequest = {
  runId: string
  model: ChatModelDescriptor
  messages: CanonicalChatMessage[]
  maxOutputTokens?: number
  signal: AbortSignal
}

type NormalizedChatStreamEvent =
  | { type: 'response-start'; providerRequestId?: string }
  | { type: 'part-start'; partId: string; kind: 'text' | 'refusal' | 'citation' | 'unknown'; providerType?: string }
  | { type: 'text-delta'; partId: string; delta: string }
  | { type: 'part-complete'; partId: string; metadata?: Record<string, unknown> }
  | { type: 'usage'; inputTokens?: number; outputTokens?: number }
  | { type: 'response-complete'; stopReason?: string }

type ChatProviderConnectionInput = {
  credential: string
  config: unknown
}

interface ChatProviderAdapter {
  readonly id: string
  readonly label: string
  readonly credentialFields: CredentialField[]
  validateConnection(input: ChatProviderConnectionInput): Promise<ChatConnectionHealth>
  parseConnectionConfig(raw: unknown): ChatProviderConfig
  listModels(input: ChatProviderConnectionInput): Promise<ChatModelDescriptor[]>
  stream(input: ChatProviderConnectionInput, request: CanonicalChatRequest): AsyncIterable<NormalizedChatStreamEvent>
  mapError(error: unknown): ChatProviderFailure
}
```

The registry rejects duplicate provider ids, missing streaming support, invalid model descriptors, and
adapters whose credential/config public projection could expose a secret. A fake provider implements
the same interface and drives most tests/E2E.

## 2. SDK decision

Use the official `openai` and `@anthropic-ai/sdk` TypeScript packages inside provider adapters.

Rationale:

- Acorn currently documents Node 20 as its minimum; the current Vercel AI SDK repository documents a
  Node 22 baseline.
- Only two providers are required now, and their official SDKs expose typed stream events, request ids,
  error classes, retries, cancellation, and current multimodal request shapes.
- OpenAI Responses and Anthropic Messages differ materially. An Acorn canonical boundary is needed
  even if a third-party unified SDK is used, so stored/UI types must not become that SDK's types.
- Direct adapters keep provider-specific options available later without leaking them into the common
  model.

Reconsider a unified generation library when provider count reaches roughly four and measured adapter
duplication exceeds the cost of the extra dependency/runtime baseline. Even then, preserve
`ChatProviderAdapter` and canonical persistence as Acorn's boundary.

## 3. OpenAI adapter

### API

- Use the Responses API, not legacy Completions.
- Set `stream: true`.
- Set `store: false` where the selected API/model supports it; Acorn is the conversation authority.
- Send locally reconstructed messages each turn rather than relying on `previous_response_id`.
- Capture the response/request id for diagnostics only.

### Request mapping

Canonical parts map as follows:

| Canonical | Responses input |
| --- | --- |
| user/assistant text | role item with `input_text`/provider-appropriate text content |
| image bytes | `input_image` with a `data:<mime>;base64,...` URL |
| PDF bytes | `input_file` with `filename` and base64 `file_data` |

Text file attachments have already been decoded by the request assembler and represented as bounded
text with an Acorn-generated delimiter, for example:

```text
<attachment name="schema.sql" media-type="text/plain">
...
</attachment>
```

The delimiter is generated by the request assembler, not stored in the message or rendered to the
user. Escape delimiter-like content inside the attachment.

### Stream mapping

At minimum handle:

- response creation/start → `response-start`;
- output item/content-part creation → `part-start`;
- `response.output_text.delta` → `text-delta`;
- refusal/citation events if exposed → canonical refusal/citation parts;
- response completion → usage plus `response-complete`;
- error/incomplete events → throw a typed mapped failure.

Unknown OpenAI event types are ignored only if they are documented as additive and do not affect
message correctness. Record a debug counter, never serialize the whole event body into logs. If an
unknown event denotes visible output, persist an `unknown` part with a bounded type/metadata summary.

### Errors

Map SDK/API failures structurally:

- authentication/permission → `chat_provider_needs_auth` and connection `needs-auth`;
- rate limit → `chat_rate_limited` with retry-after metadata kept server-side;
- model missing/unsupported → `chat_model_unavailable`;
- content/request validation → `chat_provider_rejected` or the more specific attachment/context code;
- transport/5xx → `chat_provider_unavailable`;
- user abort → `chat_cancelled`.

Never use provider message text as branching logic. It may appear only as redacted/bounded detail.

## 4. Anthropic adapter

### API

- Use the Messages API with `stream: true`.
- Reconstruct the complete prior conversation from Acorn rows each turn; the Messages API is stateless
  with respect to the local thread.
- Do not send hidden system/context material in v1. The `system` field is absent.
- Capture `request-id`/message id for diagnostics only.

### Request mapping

| Canonical | Messages content |
| --- | --- |
| user/assistant text | `text` content block |
| image bytes | `image` block with base64 source and validated media type |
| PDF bytes | `document` block with base64 PDF source |

Anthropic may combine consecutive same-role turns. The assembler should already produce valid linear
history, but the adapter must defensively merge consecutive same-role canonical messages and never
invent an empty turn.

### Stream mapping

Anthropic's stream has a message lifecycle and indexed content-block lifecycle:

- `message_start` → `response-start`;
- `content_block_start` → `part-start` keyed by provider block index;
- text deltas → `text-delta`;
- content block stop → `part-complete`;
- message delta usage/stop reason → `usage`/terminal metadata;
- `message_stop` → `response-complete`;
- provider error event → typed failure.

Handle ping and unknown additive event kinds without failing. Preserve ordered block identity even when
multiple blocks exist.

### Errors

Use the same canonical behavior as OpenAI. Preserve provider request id and status in server-only
diagnostics. Anthropic `overloaded`/5xx maps to unavailable, `429` to rate-limited, invalid key to
needs-auth, and request-too-large to attachment/context-too-large where the cause can be identified.

## 5. Model catalog

The renderer must never hardcode model ids or infer attachment support from string prefixes.

Each adapter returns a normalized model list built from:

1. remote provider model listing where available;
2. adapter-owned capability metadata for known model families/snapshots;
3. historical/custom ids already selected in Acorn, represented as `unverified` until validated.

Model list behavior:

- cache per connection in memory for 15 minutes;
- manual refresh bypasses cache;
- a fetch failure serves the last successful in-process list marked stale;
- no model list is persisted as durable truth;
- deprecated models remain visible only when already selected, with a warning;
- display/provider labels come from the adapter; ids remain opaque strings;
- a model is selectable only if it supports streamed text output;
- attachment controls consult `ChatModelCapability`.

Do not scrape pricing or calculate spend in the selector. If a future price catalog is added, it must
be dated, versioned, and clearly approximate.

## 6. Credential lifecycle

The chat settings flow is workspace-specific:

1. Renderer submits credential fields once to a workspace connection endpoint.
2. Server adapter validates with a cheapest safe request (prefer model-list/account endpoint over a
   billable generation when possible).
3. Server normalizes non-secret config and encrypts the credential with `SESSION_ENC_KEY`.
4. Renderer receives only connection id/provider/label/status/model-catalog summary.
5. Test, rotate, disable, and disconnect act on the stable connection id.

Rotation replaces encrypted credential/config on the same row. Threads/runs keep their connection id
and historical provider/model strings. Disconnect is blocked while a run is active, then soft-deletes
or removes the connection after cancellation; historical messages remain readable.

Base URL/custom OpenAI-compatible endpoints are deliberately deferred. When added, they belong to
adapter config and must pass URL/credential/SSRF validation. Do not smuggle them into `providerId` or
model id.

## 7. Canonical request assembly

Provider adapters receive a final canonical request. They do not query SQLite or Acorn services.

`ChatRequestAssembler` executes these stages:

1. Load thread, messages, parts, attachment metadata, run, selected connection, and model descriptor.
2. Validate all lineage and terminal/active status assumptions.
3. Convert stored visible parts to canonical text/image/PDF content. Decode validated text-file
   attachments into bounded text blocks with an escaped Acorn-generated filename/media-type delimiter;
   this provider-neutral transformation applies to both OpenAI and Anthropic.
4. Invoke the future context registry. In v1 it returns `[]` unconditionally.
5. Budget history and attachments for the selected model.
6. Produce and persist `ChatInputManifestV1` before provider I/O.
7. Return canonical request plus decrypted credential in a short-lived server object.

The decrypted key must not be captured in run state, event closures that outlive the request, error
metadata, or logs. Release references after the provider stream terminates.

## 8. Conversation history policy

The full local thread remains visible and durable. The model input may omit oldest turns when the
selected model's budget requires it.

Budget policy:

- reserve output budget first;
- always include the current user message and its attachments or reject before sending;
- walk earlier complete user/assistant pairs newest-first;
- never include a failed/cancelled/interrupted assistant part as authoritative history unless the user
  explicitly retries with “include partial” in a future feature;
- keep message pairs intact; do not include an assistant response without its user request;
- prefer explicit provider token counting when it does not add a second billable generation;
- otherwise use a conservative adapter estimate and let provider validation remain authoritative;
- record included and omitted message ids in the manifest;
- surface a subtle “older messages were not sent to this model” marker on the affected response.

No automatic summarization in v1. Summaries introduce a hidden model call, cost, another persisted
representation, and possible fact loss. Add them later as an explicit context contribution with
provenance.

## 9. Future context contribution

Define the seam now but register no contributors:

```ts
type ChatContextRequest = {
  workspaceId: string
  threadId: string
  userMessageId: string
  model: ChatModelDescriptor
}

type ChatContextItem = {
  contributionId: string
  sourceKind: string
  sourceId: string
  label: string
  mediaType: string
  content: { kind: 'text'; text: string } | { kind: 'object'; snapshotRef: string }
  contentHash: string
  estimatedTokens?: number
  truncated: boolean
}

interface ChatContextContribution {
  id: string
  when(request: ChatContextRequest): boolean
  collect(request: ChatContextRequest, signal: AbortSignal): Promise<ChatContextItem[]>
}
```

V1 behavior is an empty frozen registry or an assembler dependency whose `collectAll()` returns `[]`.
Do not expose UI toggles for sources that do not exist.

Future rules:

- context is opt-in and visible in the composer/run details;
- every item has provenance, size, hash, and truncation state;
- mutable sources are snapshotted or hash-addressed for reproducibility;
- collection occurs before provider mapping and respects cancellation/budgets;
- context never writes into `chat_messages` as if the user typed it;
- plugin-to-plugin access occurs through contributions/services, not imports;
- memory retrieval is one contributor, not a special provider prompt branch.

## 10. Provider contract conformance

Every adapter must pass one shared suite:

- unique stable id and valid public descriptor;
- credential validation never returns the credential;
- connection config parser rejects malformed/secret-bearing public data;
- model ids round-trip as opaque strings;
- a text stream produces ordered start/delta/complete and usage when available;
- multi-part output preserves part order;
- cancellation aborts network work and maps to `chat_cancelled`;
- auth, rate-limit, unavailable, model-missing, and request-too-large errors map canonically;
- image/PDF mapping occurs only when model capability allows it;
- unknown additive events do not crash the stream;
- logs/spies contain no authorization header, key, base64 file body, or prompt body.

The fake provider supports deterministic delayed chunks, injected errors, cancellation, multiple parts,
and usage. UI/route/E2E tests use it instead of live APIs.
